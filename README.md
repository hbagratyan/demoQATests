# DemoQA tests 4 WiseBits by hbagratyan

Привет, коллеги! 👋 
Это выполненное задание с UI и API тестами. 
Проект реализован так, чтобы быть масштабируемым, легко читаемым и поддерживаемым (без фанатизма).
Как и требовалось в задании, тестируется публичный стенд DemoQA.

---

## 🛠️ Технологический стек

- **[Playwright](https://playwright.dev/)** - мощный фреймворк для E2E и API тестирования.
- **TypeScript** - строгая типизация для предотвращения ошибок на этапе написания кода.
- **[@faker-js/faker](https://fakerjs.dev/)** - библиотека для надежной и разнообразной генерации тестовых данных (имена, email'ы, пароли и т.д.).

---

## 🏛️ Архитектура проекта

Код постарался структурировать таким образом, чтобы четко разделять логику выполнения тестов, работу с данными и взаимодействие с интерфейсами или API.

```text
tests/
 ├── api-clients/       # Классы для работы с HTTP-запросами (API)
 ├── constants/         # Константы (URL, тестовые пользователи и т.д.)
 ├── fixtures/          # Кастомные фикстуры Playwright (настройка окружения)
 ├── locators/          # Локаторы для UI элементов (на собесе я рассказывал, что предпочитаю его юзать, вот и повод показать. Хотя для такого крохотного проекта это оверкилл, безусловно.)
 ├── page-objects/      # Описание страниц интерфейса (POM)
 ├── specs/             # Собсна тесты
 └── utils/             # Вспомогательные утилиты (генератор данных)
```

### 🧩 Умные фикстуры (Playwright Fixtures)
Вся "магия" подготовки к тестам скрыта в кастомных фикстурах (`tests/fixtures`). Например, фикстура `authorizedUser` для API:
1. Автоматически создает нового пользователя.
2. Генерирует и сохраняет bearer-токен.
3. Прокидывает полностью готовый контекст прямо в тест.
Тестам больше не нужно знать, как происходит авторизация, а мы можем сконцентрироваться только на проверке бизнес-логики

## 🌐 API Тесты

**Расположение:** `tests/specs/api-tests/`

Для тестирования API выбран паттерн **API Client**.
Вместо того чтобы захламлять тесты прямыми вызовами `request.post()`, вся работа с эндпоинтами инкапсулирована в специальные клиенты: `UserClient` и `BookClient` (находятся в `tests/api-clients/`).

**Преимущества такого подхода:**
* Тесты читаются как простой текст `await bookClient.addBook(...)`
* Централизованное управление заголовками и токенами.
* Покрытие позитивных (создание, удаление) и негативных (ошибки 400, 401) сценариев.

## 🖥️ UI Тесты

**Расположение:** `tests/specs/ui-tests/`

Основа UI тестирования в этом проекте - классический и проверенный временем **Page Object Model (POM)**.
* **BasePage**: В файле `tests/page-objects/base-page.ts` реализован базовый класс, предоставляющий общие методы, такие как надежное ожидание загрузки DOM (`openWithDomWaiting`), кастомные проверки статус-кодов ответов сети и очистка кук.
* **Страницы**: Логика взаимодействия с сайтом разбита на конкретные страницы (`MainPage`, `WebTablesPage` и т.д.).

Каждое взаимодействие логируется через `test.step`, что делает отчеты Playwright максимально детализированными и понятными при разборе падений.
Это особенно хорошо смотрится в Allure отчетах, например, при дебаге тестов.

## 🚀 Как запустить?

### 1. Установка зависимостей
Убедитесь, что у вас установлен **Node.js**. Затем выполните:
```bash
npm install
npx playwright install
```

### 2. Запуск тестов
Чтобы запустить все тесты в headless-режиме (без интерфейса браузера):
```bash
npx playwright test
```

### 3. Отчеты
Playwright также умеет генерить довольно удобный HTML-отчет (шаги `test.step`, данные о прохождении). Чтобы посмотреть его:
```bash
npx playwright show-report
```
